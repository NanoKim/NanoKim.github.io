Description

본 시스템은 Keepalived 기반 Active-Standby 구조로 설계했습니다. Primary 서버 장애 발생 시 VIP가 Secondary 서버로 자동 전환되어 서비스 중단 없이 요청이 처리될 수 있도록 했습니다.

Web Server는 Nginx 기반으로 구성되어 트래픽을 처리하며, Application Layer는 Spring Boot 구조로 비즈니스 로직을 수행합니다.

데이터는 PostgreSQL과 repmgr 기반 Streaming Replication을 통해 이중화되어 장애 상황에서도 데이터 정합성이 유지되도록 설계하였습니다.

또한, Docker와 같은 컨테이너 환경 대신 사전 구성된 의존성 캐싱 방식을 적용하여 서버 구축 시 패키지 설치 시간을 최소화하고, USB 기반 자동 구축 환경을 통해 빠르고 일관된 운영 환경 구성이 가능하도록 하였습니다.

  • Keepalived 기반 VIP Failover 구성
  • Nginx 기반 Reverse Proxy 처리 구조
  • PostgreSQL + repmgr 기반 이중화 및 Streaming Replication
  • 의존성 캐싱 기반 표준화된 실행 환경 구성
  • USB 기반 자동화된 서버 구축 프로세스

  • 소스 코드 및 형상관리 환경을 USB에 포함하여 폐쇄망에서도 동일한 개발/운영 환경 유지 가능

Result
  • 수동 복구 과정이 제거되어 장애 대응 시간이 획기적으로 단축

  • 운영 안정성과 신뢰성이 크게 향상

/**
 * MiddleApi
 *
 * - RESTful API 통신을 위한 공통 레이어
 * - CSRF 처리, 요청 옵션 구성, 에러 핸들링을 일관되게 관리
 * - 모든 API 호출은 해당 클래스를 통해 수행되도록 설계
 *
 * @example
 * const api = new MiddleApi(optionBuilder, csrf)
 * const result = await api.post('/user', payload)
 */
export class MiddleApi {
  private readonly optionBuilder: OptionBuilder;
  private readonly csrf: Csrf;

  constructor(optionBuilder: OptionBuilder, csrf: Csrf) {
    this.optionBuilder = optionBuilder;
    this.csrf = csrf;
  }

  /**
   * 공통 API 요청 처리
   *
   * - 모든 HTTP 요청(GET, POST, PUT, DELETE)의 진입점
   * - 요청 전 CSRF 토큰 세팅
   * - 공통 에러 핸들링
   *
   * @param url 요청 URL
   * @param options fetch 옵션 (method, body, query 등)
   * @returns Promise 제네릭 응답 타입
   */
  private async request(
    url: string,
    options: ReturnType
  ): Promise {
    await this.csrf(this.optionBuilder);

    try {
      return await $fetch(url, options);
    } catch (error) {
      this.handleError(error, url, options);
    }
  }

  /**
   * 공통 에러 처리
   *
   * @param error 발생한 에러 객체
   * @param url 요청 URL
   * @param options 요청 옵션
   * @throws 항상 에러를 다시 throw (상위에서 추가 처리 가능)
   */
  private handleError(
    error: unknown,
    url: string,
    options: ReturnType
  ): never {
    Common.toast.show({
      message: '거래 처리 중 오류가 발생했습니다.',
      type: 'error'
    });

    switch (process.env.NODE_ENV) {
      case 'local' : 
      case 'dev' : 
      case 'test' : 
        console.error('[API ERROR]', error);
        break;
      case 'prod' :
        Common.logger.send({
          level: 'error',
          message: (error as Error).message,
          stack: (error as Error).stack,
          url,
          options,
          env: process.env.NODE_ENV
       });
        break;
    }

    throw error;
  }

  /**
   * GET 요청
   *
   * - 조회성 데이터 요청
   * - query parameter를 통해 데이터 전달
   *
   * @param url 요청 URL
   * @param params query parameters
   */
  public get(url: string, params?: RequestData): Promise {
    return this.request(
      url,
      this.optionBuilder({
        method: 'GET',
        query: params
      })
    );
  }

  /**
   * POST 요청
   *
   * - 리소스 생성 및 일반 데이터 전송
   * - body에 JSON 형태로 데이터 전달
   *
   * @param url 요청 URL
   * @param data 요청 body 데이터
   */
  public post(url: string, data?: RequestData): Promise> {
    return this.request>(
      url,
      this.optionBuilder({
        method: 'POST',
        body: prepareRequest(data)
      })
    );
  }

  /**
   * PUT 요청
   *
   * - 리소스 전체 수정
   *
   * @param url 요청 URL
   * @param data 요청 body 데이터
   */
  public put(url: string, data?: RequestData): Promise> {
    return this.request>(
      url,
      this.optionBuilder({
        method: 'PUT',
        body: prepareRequest(data)
      })
    );
  }

  /**
   * DELETE 요청
   *
   * - 리소스 삭제
   * - query parameter 기반 전달
   *
   * @param url 요청 URL
   * @param params query parameters
   */
  public delete(url: string, params?: RequestData): Promise> {
    return this.request>(
      url,
      this.optionBuilder({
        method: 'DELETE',
        query: params
      })
    );
  }

  /**
   * 파일 업로드 요청
   *
   * - multipart/form-data 방식
   * - 사용자 정보 기반 헤더 추가
   *
   * @param url 요청 URL
   * @param data 파일 데이터
   */
  public filePost(url: string, data: RequestData): Promise> {
    const userInfo = getUserInfo();

    const headers = userInfo?.employeeNo
      ? { 'X-KKB-TLRNO': userInfo.employeeNo }
      : undefined;

    return this.request>(
      url,
      this.optionBuilder({
        method: 'POST',
        body: prepareFormData(data),
        headers
      })
    );
  }

  /**
   * 마스킹 처리 요청
   *
   * - 요청 데이터에 contextCustomizer 적용
   * - 민감 데이터 마스킹 또는 변환 처리
   *
   * @param url 요청 URL
   * @param data 요청 데이터
   * @param contextCustomizer 컨텍스트 커스터마이저
   */
  public masking(
    url: string,
    data: RequestData,
    contextCustomizer: (context: SmurfContext) => void = () => {}
  ): Promise> {
    return this.request>(
      url,
      this.optionBuilder({
        method: 'POST',
        body: prepareRequest(data, contextCustomizer)
      })
    );
  }

  /**
   * 로그인 요청
   *
   * - 프로젝트 코드 기반 로그인 URL 구성
   * - 인증 처리 수행
   *
   * @param payload 로그인 요청 데이터
   */
  public async login(payload: RequestData): Promise {
    const config = useRuntimeConfig();
    const url = `/login/${config.public.projectCode}`;

    return this.request(
      url,
      this.optionBuilder({
        baseURL: this.optionBuilder().rawUrl,
        method: 'POST',
        body: payload
      })
    );
  }

  /**
   * 사용자 정보 조회
   */
  public user(): Promise {
    return this.request(
      '/user',
      this.optionBuilder({
        method: 'GET'
      })
    );
  }

  /**
   * 사용자 권한 조회
   */
  public role(): Promise {
    return this.request(
      '/role',
      this.optionBuilder({
        method: 'GET'
      })
    );
  }

왜 MiddleApi를 따로 만들었나요?
단순히 fetch를 사용하는 것이 아니라,
모든 API 요청을 하나의 흐름으로 통제하기 위해 설계했습니다.

프로젝트를 진행하면서 API 호출 방식이 제각각 흩어지는 문제가 있었고,
이를 해결하기 위해 단일 진입점(request) 구조로 통합했습니다.
이 구조의 장점은 무엇인가요?
API 호출을 단순 함수가 아니라 하나의 레이어로 관리할 수 있습니다.

새로운 기능이 추가되더라도
기존 코드 수정 없이 확장 가능한 구조를 유지할 수 있습니다.
설계하면서 가장 중요하게 본 포인트는?
단순 구현이 아니라 "구조"를 만드는 것에 집중했습니다.

  • 단일 책임 → 요청 흐름만 담당
  • 확장성 → optionBuilder 분리
  • 환경 분리 → dev / prod 로깅 전략 분리
  • 타입 안정성 → 제네릭 기반 응답 처리
이 구조에서 해결하고자 한 문제는 무엇인가요?
가장 큰 문제는 중복과 일관성 부족이었습니다.

  • CSRF 처리 위치가 제각각
  • 에러 핸들링 방식이 다름
  • 요청 옵션 구성 방식이 통일되지 않음

이를 해결하기 위해 공통 로직을 내부로 캡슐화했습니다.
왜 request 메서드 하나로 통합했나요?
GET, POST, PUT, DELETE 요청이 각각 흩어져 있으면
공통 로직이 중복되고 관리 포인트가 분산됩니다.

그래서 모든 요청을 단일 진입점(request)으로 모아
CSRF, 에러 처리, 옵션 구성을 한 곳에서 통제하도록 설계했습니다.

결과적으로 코드 중복을 제거하고, 유지보수 비용을 크게 줄일 수 있었습니다.
이 코드를 통해 보여주고 싶은 점은?
저는 단순히 기능을 구현하는 개발자가 아니라,
반복되는 문제를 구조로 해결하는 개발자입니다.

MiddleApi는 그 고민의 결과물입니다. 

package com.example.asset.service;

import com.example.asset.dto.AssetRequest;
import com.example.asset.dto.AssetResponse;
import com.example.asset.entity.Asset;
import com.example.asset.repository.AssetRepository;
import com.example.asset.external.ExternalValuationClient;

import com.example.api.util.exception.BusinessErrorException;
import com.example.api.util.exception.code.CommonBusinessErrorExceptionCode;

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

import java.math.BigDecimal;

/**
 * AssetService
 *
 * - 유무형 자산 등록 및 평가 처리 서비스
 * - Validation / Calculation / Save / External Call 단계 분리
 * - 메인 흐름 중심의 가독성 확보
 */
@Slf4j
@Service
@RequiredArgsConstructor
public class AssetService {

    private final AssetRepository assetRepository;
    private final ExternalValuationClient externalClient;

    /**
     * 자산 등록 및 평가 처리
     *
     * @param request 자산 요청 DTO
     * @return AssetResponse 처리 결과
     */
    @Transactional
    public AssetResponse processAsset(AssetRequest request) {

        log.info("=======================================");
        log.info(" [ASSET PROCESS START]");
        log.info("=======================================");
        log.info("▶ 요청 자산명: {}", request.getAssetName());
        log.info("▶ 요청 금액: {}", request.getAmount());

        // STEP 1. Validation
        validate(request);

        // STEP 2. Calculation
        BigDecimal evaluatedAmount = calculate(request);

        // STEP 3. Entity 생성
        Asset asset = createEntity(request, evaluatedAmount);

        // STEP 4. Save
        Asset savedAsset = save(asset);

        // STEP 5. External Call
        callExternal(savedAsset);

        log.info("=======================================");
        log.info(" [ASSET PROCESS END]");
        log.info("▶ 저장 ID: {}", savedAsset.getId());
        log.info("=======================================");

        return toResponse(savedAsset);
    }

    /**
     * STEP 1. Validation
     */
    private void validate(AssetRequest request) {
        log.info("-------------------------------");
        log.info("[STEP 1] Validation");
        log.info("-------------------------------");

        if (request.getAssetName() == null || request.getAssetName().isBlank()) {
            log.error("❌ 자산명이 비어있습니다.");
            throw new BusinessErrorException(
                CommonBusinessErrorExceptionCode.DATA_REQUIRED_ASSET_NAME
            );
        }

        if (request.getAmount() == null) {
            log.error("❌ 금액이 존재하지 않습니다.");
            throw new BusinessErrorException(
                CommonBusinessErrorExceptionCode.DATA_REQUIRED_AMOUNT
            );
        }

        if (request.getAmount().compareTo(BigDecimal.ZERO) <= 0) {
            log.error("❌ 금액이 0 이하입니다.");
            throw new BusinessErrorException(
                CommonBusinessErrorExceptionCode.DATA_REQUIRED_MIN_PRICE
            );
        }

        log.info("✔ Validation 통과");
    }

    /**
     * STEP 2. Calculation
     * - 감가상각 및 평가금액 계산
     */
    private BigDecimal calculate(AssetRequest request) {
        log.info("-------------------------------");
        log.info("[STEP 2] Calculation");
        log.info("-------------------------------");

        BigDecimal baseAmount = request.getAmount();

        // 예시: 감가상각 10%
        BigDecimal depreciationRate = new BigDecimal("0.1");
        BigDecimal depreciation = baseAmount.multiply(depreciationRate);

        BigDecimal evaluated = baseAmount.subtract(depreciation);

        log.info("✔ 원금: {}", baseAmount);
        log.info("✔ 감가상각: {}", depreciation);
        log.info("✔ 평가금액: {}", evaluated);

        return evaluated;
    }

    /**
     * STEP 3. Entity 생성
     */
    private Asset createEntity(AssetRequest request, BigDecimal evaluatedAmount) {
        log.info("-------------------------------");
        log.info("[STEP 3] Entity 생성");
        log.info("-------------------------------");

        Asset asset = Asset.builder()
                .assetName(request.getAssetName())
                .amount(request.getAmount())
                .evaluatedAmount(evaluatedAmount)
                .assetType(request.getAssetType())
                .build();

        log.info("✔ Entity 생성 완료");
        return asset;
    }

    /**
     * STEP 4. Save
     */
    private Asset save(Asset asset) {
        log.info("-------------------------------");
        log.info("[STEP 4] Save");
        log.info("-------------------------------");

        Asset saved = assetRepository.save(asset);

        log.info("✔ DB 저장 완료: {}", saved.getId());
        return saved;
    }

    /**
     * STEP 5. External Call
     * - 외부 평가 시스템 연동 (Mock)
     */
    private void callExternal(Asset asset) {
        log.info("-------------------------------");
        log.info("[STEP 5] External Call");
        log.info("-------------------------------");

        try {
            externalClient.send(asset);
            log.info("✔ 외부 시스템 연동 성공");
        } catch (Exception e) {
            log.error("❌ 외부 연동 실패", e);

            // 외부 연동 실패도 비즈니스 에러로 관리
            throw new BusinessErrorException(
                CommonBusinessErrorExceptionCode.EXTERNAL_API_ERROR
            );
        }
    }

    /**
     * Response 변환
     */
    private AssetResponse toResponse(Asset asset) {
        return AssetResponse.builder()
                .id(asset.getId())
                .assetName(asset.getAssetName())
                .amount(asset.getAmount())
                .evaluatedAmount(asset.getEvaluatedAmount())
                .build();
    }
}
왜 Service 로직을 이렇게 단계별로 분리했나요?
기존에는 Validation, 계산, 저장 로직이 하나의 메서드에 혼재되어 있어
가독성과 유지보수성이 크게 떨어지는 문제가 있었습니다.

이를 해결하기 위해 Validation / Calculation / Save / External Call 단위로 분리하고,
메인 흐름만 남기는 구조로 개선했습니다.
이 구조의 핵심 장점은 무엇인가요?
가장 큰 장점은 “읽히는 코드”입니다.

processAsset 메서드만 봐도 전체 흐름이 한눈에 보이고,
각 단계는 별도의 메서드로 분리되어 있어
수정 범위와 영향도를 명확하게 통제할 수 있습니다.
왜 Validation을 따로 분리했나요?
Validation은 비즈니스 로직과 분리되어야
재사용성과 테스트 용이성을 확보할 수 있습니다.

또한 Fail Fast 전략을 적용하여
잘못된 요청은 초기에 즉시 차단하도록 설계했습니다.
IllegalArgumentException 대신 BusinessErrorException을 사용한 이유는?
단순 예외가 아닌 비즈니스 에러를 구조적으로 관리하기 위함입니다.

에러 코드를 Enum으로 관리하여
프론트와의 계약(Contract)을 유지할 수 있고,
메시지 표준화 및 국제화에도 대응할 수 있도록 설계했습니다.
로그를 이렇게 상세하게 남긴 이유는 무엇인가요?
운영 환경에서는 로그가 유일한 디버깅 수단이 되는 경우가 많습니다.

STEP 단위로 로그를 남겨
어느 단계에서 문제가 발생했는지 즉시 파악할 수 있도록 했습니다.
이 코드를 통해 보여주고 싶은 점은 무엇인가요?
저는 단순히 기능을 구현하는 것이 아니라,
비즈니스 로직을 구조적으로 설계하는 개발자입니다.

반복되는 문제를 코드 레벨이 아닌
구조 레벨에서 해결하는 데 집중하고 있습니다. 

WITH RECURSIVE org_tree AS (

  /* =========================================================
   * 1. 루트 조직 조회 (최상위 노드)
   * ========================================================= */
  SELECT
         o.org_id                                               /* 조직 ID */
       , o.parent_id                                            /* 상위 조직 ID */
       , o.org_nm                                               /* 조직명 */
       , 1 AS depth                                             /* 계층 깊이 */
       , LPAD(o.org_id::TEXT, 5, '0')                   AS path /* 정렬용 경로 */

    FROM tb_org o
   WHERE o.parent_id IS NULL                                    /* ROOT NODE */

   UNION ALL

  /* =========================================================
   * 2. 하위 조직 재귀 조회
   *    - 부모 → 자식으로 계층 확장
   *    - depth 증가 및 path 누적
   * ========================================================= */
  SELECT
         c.org_id                                               /* 조직 ID */
       , c.parent_id                                            /* 상위 조직 ID */
       , c.org_nm                                               /* 조직명 */
       , p.depth + 1 AS depth                                   /* 계층 깊이 증가 */
       , p.path || '>' || LPAD(c.org_id::TEXT, 5, '0')  AS path /* 경로 누적 */

    FROM tb_org c
    JOIN org_tree p
      ON c.parent_id = p.org_id

)

/* =========================================================
 * 3. 최종 트리 조회
 *    - 계층 구조를 평탄화(Flatten)하여 조회
 *    - path 기반 정렬로 트리 구조 유지
 * ========================================================= */
SELECT
       t.org_id                                                 /* 조직 ID */
     , t.parent_id                                              /* 상위 조직 ID */
     , t.org_nm                                                 /* 조직명 */
     , t.depth                                                  /* 계층 깊이 */
     , t.path                                                   /* 전체 경로 */

  FROM org_tree t

/* =========================================================
 * 4. 특정 조직 기준 하위 트리 조회 (선택적 사용)
 *    - :target_org_id 바인딩 시 subtree 조회 가능
 * ========================================================= */
-- WHERE t.path LIKE (
--     SELECT p.path || '%'
--       FROM org_tree p
--      WHERE p.org_id = :target_org_id
-- )

/* =========================================================
 * 5. 정렬
 *    - path 기준 정렬 → 계층 구조 유지
 * ========================================================= */
  ORDER BY t.path
왜 Recursive CTE를 사용했나요?
기존에는 계층 깊이에 따라 쿼리가 반복 호출되는 N+1 문제가 발생하고 있었습니다.

이를 해결하기 위해 Recursive CTE를 활용하여 단일 쿼리로 트리 전체를 조회하도록 개선했습니다.
Path 컬럼을 따로 만든 이유는 무엇인가요?
단순 depth 기반 정렬은 계층 구조를 보장하지 못하기 때문에,
정렬과 계층 표현을 동시에 해결하기 위해 Path 기반 구조를 도입했습니다.

이를 통해 트리 형태를 유지하면서 안정적으로 정렬할 수 있습니다.
이 쿼리에서 성능 개선 포인트는 무엇인가요?
가장 큰 개선은 반복적인 재귀 호출 제거입니다.

기존에는 계층 깊이만큼 쿼리가 실행되었지만,
현재는 단일 쿼리로 전체 트리를 구성하여 DB 부하를 크게 줄였습니다.
이 구조의 확장성은 어떻게 확보했나요?
depth와 path를 함께 제공함으로써 다양한 활용이 가능하도록 설계했습니다.

  • 특정 조직 기준 subtree 조회 가능
  • UI 트리 구조 즉시 렌더링 가능
  • 계층 이동 및 영향 범위 추적 가능
왜 PostgreSQL 기반으로 작성했나요?
표준 SQL 기반의 Recursive CTE 지원과 문자열 처리 유연성을 고려했습니다.

특정 DB에 종속되지 않고, 확장성과 이식성을 고려한 SQL 스타일을 유지하기 위함입니다.
이 쿼리를 통해 보여주고 싶은 점은 무엇인가요?
단순히 데이터를 조회하는 것이 아니라,
계층형 데이터를 구조적으로 이해하고 해결할 수 있는 능력을 보여주고자 했습니다.

왜 별도의 BUILD 스크립트를 만들었나요?
단순히 수동으로 서버를 구성하는 방식은
환경마다 결과가 달라지고, 반복 작업이 많아지는 문제가 있었습니다.

이를 해결하기 위해 하나의 실행 파일(.sh)로 모든 구축 과정을 표준화했고,
누구든 동일한 결과를 얻을 수 있도록 설계했습니다.
STEP 0에서 사전 검증을 넣은 이유는 무엇인가요?
오프라인 환경에서는 실행 중 실패 = 복구 비용 증가로 이어집니다.

따라서 실행 전에 모든 필수 파일을 검증하여
문제가 있으면 즉시 중단하도록 설계했습니다.

이는 Fail Fast 전략을 적용한 것입니다.
왜 echo 로그를 이렇게 상세하게 남겼나요?
폐쇄망 환경에서는 디버깅이 제한적이기 때문에
로그 자체가 유일한 추적 수단이 됩니다.

단계별 로그를 통해 현재 진행 상태를 명확히 하고,
문제 발생 시 어디서 실패했는지 빠르게 파악할 수 있도록 했습니다.
Gradle과 npm을 offline 모드로 구성한 이유는?
외부 네트워크 없이도 빌드가 가능하도록 하기 위해
의존성을 사전에 캐싱하고 offline 모드를 사용했습니다.

이를 통해 인터넷 연결 없이도 동일한 빌드 결과를 보장합니다.
왜 '실행'이 아니라 '실행 준비 완료'로 끝냈나요?
실제 운영 환경에서는 실행 시점이 분리되는 경우가 많기 때문에,
빌드와 실행 단계를 명확히 분리했습니다.

이를 통해 배포/운영 프로세스를 유연하게 제어할 수 있도록 했습니다.
이 스크립트를 통해 보여주고 싶은 점은?
저는 단순히 기능을 구현하는 것을 넘어서,
